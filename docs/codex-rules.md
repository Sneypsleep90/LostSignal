# Lost Signal — Codex Rules

## Назначение

Этот файл фиксирует правила работы Codex внутри проекта **Lost Signal**.

Codex должен воспринимать Lost Signal не как обычный habit tracker, а как премиальный интерфейс-путь для ухода от цифрового шума.

Главная идея проекта:

> Уйти от шума.

---

# Перед началом работы

Перед любыми изменениями в проекте сначала прочитать:

```txt
design/visual-language.md
docs/product-vision.md
```

Эти файлы являются главным контекстом проекта.

Если задача противоречит этим файлам, сначала сохранить идею Lost Signal, а не просто выполнить задачу буквально.

---

# Главные правила

1. Не превращать Lost Signal в habit tracker.
2. Не добавлять мотивационный тон.
3. Не использовать слова из запрещённого списка в UI.
4. Не ломать визуальный стиль проекта.
5. Не добавлять новые библиотеки без явной необходимости.
6. Не переписывать весь проект, если задача локальная.
7. Не удалять существующую рабочую логику без причины.
8. Не усложнять архитектуру.
9. Не добавлять геймификацию.
10. Не делать интерфейс ярче, шумнее или плотнее, чем нужно.

---

# Продуктовая суть

Lost Signal — это не приложение привычек.

Пользователь не:

* закрывает цель;
* держит стрик;
* проходит челлендж;
* получает достижения;
* прокачивает продуктивность.

Пользователь:

* уходит от шума;
* увеличивает дистанцию;
* возвращает себе время;
* движется глубже по маршруту;
* видит, как сигнал становится слабее.

Визуальное направление:

* тёмный северный лес;

* стеклянные карточки;

* тяжёлые тени;

* декоративные элементы без смысла.

# Продуктовая модель

Основная модель приложения:

```js
signalProfile
```

Не возвращаться к старой модели, где хранится только дата.

Обязательные поля:

```js
{
  startDate: "2026-05-15",
  target: "short_videos",
  dailyUsageMinutes: 90,
  goalDays: 30,
  theme: "noisy_forest"
}
```

## Значение полей

### `startDate`

Дата, когда пользователь вышел из шума.

### `target`

Что пользователь оставил позади.

Примеры:

```js
"short_videos"
"social_media"
"news"
"games"
"custom"
```

### `dailyUsageMinutes`

Сколько времени шум забирал в день.

Используется для расчёта возвращённого времени.

### `goalDays`

Первый рубеж пути.

Не называть это “целью” в интерфейсе.

### `theme`

Визуальный маршрут.

На MVP основная тема:

```js
"noisy_forest"
```

UI label:

```txt
Шумный лес
```

# localStorage

Использовать один основной ключ:

```js
lost_signal_profile
```

Не плодить лишние localStorage keys без необходимости.

Если в проекте есть старые данные, например только дата старта, нужно сделать безопасную миграцию в `signalProfile`.

Fallback-значения:

```js
{
  target: "short_videos",
  dailyUsageMinutes: 90,
  goalDays: 30,
  theme: "noisy_forest"
}
```

# Работа с задачами

## Если задача про данные

Не переписывать визуальный UI без необходимости.

Пример:

* storage;
* constants;
* migration;
* calculations.

Делать минимальный diff.

---

## Если задача про UI

Не менять модель данных без необходимости.

Пример:

* dashboard layout;
* onboarding screens;
* settings visual;
* journey progress.

Сохранять визуальный язык из `design/visual-language.md`.

---

## Если задача про onboarding

Не ломать dashboard.

Onboarding должен:

* быть fullscreen;
* не быть popup;
* задавать один вопрос на экран;
* сохранять `signalProfile`;
* вести пользователя к dashboard.

---

## Если задача про dashboard

Не ломать onboarding.

Dashboard должен показывать:

* дни вдали от шума;
* что осталось позади;
* возвращённое время;
* текущую локацию;
* дату начала;
* первый рубеж;
* journey progress.

---

## Если задача про settings

Не превращать настройки в центр продукта.

Настройки должны быть минимальными:

* что оставлено позади;
* время в день;
* дата начала;
* первый рубеж;
* маршрут;
* начать заново.

---


1. Не появился ли habit tracker tone?
2. Не используются ли запрещённые слова в UI?
3. Не сломан ли `signalProfile`?
4. Не стал ли интерфейс визуально шумнее?
5. Не добавлена ли лишняя библиотека?
